Introduction

Designing and building my homelab has been one of the most rewarding projects I’ve taken on and the reasons behind it are twofold. First it’s a perfect sandbox for learning about networking, especially advanced topics like network segregation, VLANs and subnetting. In a world where everything is connected and secruity is crucial, having the ability to properly segment networks is a valuable skill.

Second I wanted a reliable environment to experiment with and run Docker containers which are essential in modern development and deployment. On top of that, having my own homelab means I no longer have to pay for certain cloud-based services as I can now host many of these solutions myself. By building this homelab, I not only gained hands-on experience but also saved money while learning at my own pace.

Network

When building out my homelab, I chose Ubiquiti network gear for its balance of enterprise-level features, ease of use, and cost-effectiveness. The cornerstone of my setup si the Ubiquiti Dream Machine Pro (UDM Pro), complemented by a USW 24 Pro PoE switch, several other PoE switches around the house and NanoHD APs for WiFi.

One of the biggest reasons I went with Ubiquiti is its intuitive UniFi interface, which makes managing complex networking tasks like VLAN configuration and traffic management, straightforward. The ability to scale the network and integrate everything from switches, APs to cameras from a single interface with ease was also a major factor in my decision.

With this setup, I created separate VLANs to segment my network for better security and organization. My main network handles everyday traffic, while I have a dedicated IoT and security network. This segregation is essential for keeping potentially vulnerable IoT devices isolated from critical infrastructure, minimizing the risk of breaches. An interesting challenge I faced was setting up my ISP’s set-top box, which required a direct IP from the ISP to function properly. To resolve this, I had to run a second cable form my modem to the switch and configure a VLAN specifically for the set-top box, allowing it to bypass my UDM Pro and get an IP directly from the ISP.

In addition to my Ubiquiti gear, I’ve integrated other critical components into my network. I use a Synology NAS for storage, backup and file sharing. For home automation, I run Home Assistant on a separate Raspberry Pi 4, which handles everything from lighting to smart appliances. With the Ubiquiti system providing a strong and flexible backbone, I’ve been able to build a network that supports these diverse applications and devices efficiently while keeping everything secure.

Applications

When deciding how to run applications within my homelab, I chose Proxmox with LXC containers for running Docker, and there a few key reasons for my choice.

Proxmox offers a robust and user-friendly virtualization platform that supports both full virtual machines (VMs) and lightweight LXC containers, providing flexibility in how I manage workloads. This flexibility is useful because I need both isolated environments and efficient resource usage.

LXC containers are lightweight and consume fewer resources compared to full VMs which is ideal for running Docker. Since Docker itself is container-based, running it inside an LXC container allows for maximum efficiency with minimal overhead. this setup helps me get the most out of my hardware, running multiple services or environments without taxing my system.

Additionally Proxmox’s integration with LXC makes it easy to manage containers, snapshots, backups and networking, all through a powerful and easy to use web interface.

Stacks

In my homelab, I’ve set up a mix of container stacks to suit different needs. For certain use case, like media or tools I run stacks that contain multiple containers working together for a combined purpose. For example, my media stack includes containers for handling media servers, downloaders, and automation tools all integrated to provide a seamless experience. However for more specialized tasks, I also use LXC containers to run either a single container (or stack with only a single container). This approach allows me to keep resource usage efficient while maintaining isolation and simplicty for services that don’t need to be part of a larger ecosystem.
